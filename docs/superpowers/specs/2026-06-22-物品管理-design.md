# 物品管理（资产台账）设计说明

## 目标

在「成本管理」分组下新增一个**物品管理**能力：把团队的实物资产和虚拟商品集中成一份台账，并把每个物品同时关联到它的**成本**（一条支出记录）和**放置位置**（场地布置里的某层楼 + 某个区域）。

核心业务规则：

> 任何一个物品，除非是**虚拟商品**，否则**必须**关联一个成本，以及一个放置的地方。

为支撑"放置位置"是可靠的服务端关联，本设计还包含一个前置改造：把现有的**场地布置**从浏览器 localStorage 迁移进数据库，成为一个全团队共享的场地。

非目标（本期不做）：物品的折旧/财务摊销、领用借还审批流、扫码盘点、多物理仓库的库存调拨、物品数量与支出数量的强一致校验。

## 范围与分期

整个功能横跨三层，其中"场地入库"是其余部分的前置条件。采用**一份设计文档、分 3 期实现**，每期可独立验收：

- **第 0 期 — 场地布置入库**：新建 `venues / venue_floors / venue_items` 三张表，场地页改为读写数据库；localStorage 降级为一次性导入 + 离线缓存。本期本质是把现有功能做一次数据迁移，是最重的一期。
- **第 1 期 — 物品台账**：新建 `items` 表 + `item_status_logs` 表 + `item_code_counters` 计数表；在「成本管理」分组下新增**物品管理**页面；实物/虚拟规则在数据库约束和表单里双重校验。
- **第 2 期 — 联动展示 + 照片**：场地画布选中区域时展示该区域的物品及总成本；新增照片上传（需要新建 Supabase Storage 存储桶，项目当前没有任何上传基础设施）。

## 第 0 期：场地布置入库

### 现状

- 场地布局目前只存在浏览器 localStorage（key `guild-venue:layout:v1`），无服务端持久化。
- 类型定义在 `src/venue/layoutData.ts`：`VenueLayout` / `VenueFloor` / `VenueItem`。
- 图层顺序（z-order）目前用数组下标表达，通过 `moveVenueItemLayer` 调整。

### 数据库表（迁移文件 `029_venue.sql`）

镜像现有类型，所有表带 `created_at` / `updated_at`（沿用项目里的 updated_at 触发器写法）。

```
venues
  id            uuid PK
  name          text NOT NULL
  width         integer NOT NULL
  height        integer NOT NULL
  created_at, updated_at

venue_floors
  id              uuid PK
  venue_id        uuid NOT NULL REFERENCES venues(id) ON DELETE CASCADE
  name            text NOT NULL
  width           integer NOT NULL
  height          integer NOT NULL
  background_image text NULL
  sort_order      integer NOT NULL DEFAULT 0
  created_at, updated_at

venue_items
  id          uuid PK
  floor_id    uuid NOT NULL REFERENCES venue_floors(id) ON DELETE CASCADE
  type        text NOT NULL CHECK (type IN
              ('equipment','renovation','area','corridor',
               'door_inward','door_sliding','fire','power','network'))
  name        text NOT NULL
  x           integer NOT NULL
  y           integer NOT NULL
  width       integer NOT NULL
  height      integer NOT NULL
  rotation    numeric NOT NULL DEFAULT 0
  status      text NOT NULL CHECK (status IN
              ('planned','in_progress','completed','maintenance'))
  note        text NOT NULL DEFAULT ''
  z_index     integer NOT NULL DEFAULT 0   -- 取代数组下标的图层顺序
  created_at, updated_at
```

- 索引：`venue_floors(venue_id, sort_order)`、`venue_items(floor_id, z_index)`。
- 迁移内**预置一行 `venues`** 作为共享场地，并按 `DEFAULT_VENUE_LAYOUT` 预置默认楼层与初始对象，保证空库首次访问即有内容。

### API 与服务层

- `GET /api/venue` → 返回嵌套布局（venue → floors → items），结构与现有 `VenueLayout` 对齐，前端基本无需改造渲染逻辑。
- `PUT /api/venue` → 接收整份布局，服务端做 **upsert + diff**：新增/更新存在的楼层与对象，删除请求中已不存在的（按 id 比对）。
- 服务层 `src/lib/venue/service.ts`，沿用 expenses 的 `ServiceResult<T>`（`{ data, error: null } | { data: null, error }`）和错误码（invalid_input / not_found / forbidden / db_error）。
- 取舍说明：整份 upsert 实现简单、贴近现有"保存整份布局"的心智模型，对小团队足够；缺点是每次保存传输整份布局。后续若性能不够再拆细粒度对象接口。

### 前端改造

- 场地页 `src/app/[locale]/(app)/guild-venue/page.tsx`：
  - 初次加载从 `GET /api/venue` 拉取，渲染前显示加载态。
  - 编辑后**防抖（约 800ms）自动保存**到 `PUT /api/venue`，复用现有的保存状态 Pill（已保存 / 失败）。
  - 撤销/重做仍保留在内存（不变）。
  - localStorage 改为：(1) 一次性导入——如本浏览器存在旧布局且数据库仍是默认内容，提供"从本浏览器导入"动作；(2) 离线缓存兜底。
- 删除区域行为变化：当某区域已被物品占用时，删除会被服务端拦截（见第 1 期 RESTRICT），画布需捕获该错误并友好提示"该区域仍有物品，请先移除或改挂"。

### 共享与权限

- 单一共享场地（一行 `venues`），全团队读写同一份布局，物品放置位置对所有人含义一致。
- 沿用现有 `authGuard()` 鉴权；本期不引入逐对象的细粒度权限。

## 第 1 期：物品台账

### `items` 表（迁移文件 `030_items.sql`）

```
items
  id                       uuid PK
  item_code                text NOT NULL UNIQUE      -- 人类可读自动编号，见下
  name                     text NOT NULL
  kind                     text NOT NULL CHECK (kind IN ('physical','virtual'))
  expense_id               uuid NULL REFERENCES expenses(id) ON DELETE RESTRICT
  placement_venue_item_id  uuid NULL REFERENCES venue_items(id) ON DELETE RESTRICT
  quantity                 integer NOT NULL DEFAULT 1 CHECK (quantity >= 1)
  status                   text NOT NULL DEFAULT 'in_use' CHECK (status IN
                           ('in_use','in_storage','under_repair','disposed'))
  responsible_person       text NULL
  serial_number            text NULL                 -- 序列号 / 资产标签（人工填）
  photo_url                text NULL                 -- 第 2 期启用
  notes                    text NULL
  created_by_user_id       uuid NULL
  created_at, updated_at

  -- 核心业务规则（数据库兜底）
  CONSTRAINT items_kind_links_chk CHECK (
    (kind = 'physical' AND expense_id IS NOT NULL AND placement_venue_item_id IS NOT NULL)
    OR
    (kind = 'virtual' AND placement_venue_item_id IS NULL)
  )
```

说明：

- **实物（physical）**：`expense_id`（成本）和 `placement_venue_item_id`（放置位置）都必填。
- **虚拟（virtual）**：不允许有放置位置（`placement_venue_item_id` 必须为空）；`expense_id` 选填。
- 一个物品 = 一个放置位置。一次采购分摊到多个区域时建多条物品，多条可引用同一 `expense_id`。
- 本期**不**强制"同一支出下各物品数量之和 = 支出数量"，仅在第 2 期考虑做软提示。
- 索引：`items(kind)`、`items(status)`、`items(expense_id)`、`items(placement_venue_item_id)`、`items(item_code)`（UNIQUE 自带）。

### 人类可读编号 `item_code`

- 格式 **`WP-YYYY-NNNN`**，例如 `WP-2026-0001`，4 位零填充，**每年从 0001 重新计数**。前缀 `WP` 为常量（易改）。
- 并发安全的生成方式：

```
item_code_counters
  year      integer PK
  last_seq  integer NOT NULL DEFAULT 0
```

  `items` 上挂一个 `BEFORE INSERT` 触发器：取当前年份，对 `item_code_counters` 当年行做 `INSERT ... ON CONFLICT (year) DO UPDATE SET last_seq = last_seq + 1 RETURNING last_seq`（行级锁保证不重号），再拼成 `WP-<year>-<lpad(seq,4)>` 写入 `NEW.item_code`。
- `item_code` 只读，列表和详情显著展示，用于线下/口头指代；内部 `id`(UUID) 仅用于外键关联。

### 完整状态变更日志 `item_status_logs`（同迁移文件）

```
item_status_logs
  id                  uuid PK
  item_id             uuid NOT NULL REFERENCES items(id) ON DELETE CASCADE
  from_status         text NULL              -- 首次为 null
  to_status           text NOT NULL
  note                text NULL
  changed_by_user_id  uuid NULL
  changed_at          timestamptz NOT NULL DEFAULT now()
```

- 索引：`item_status_logs(item_id, changed_at)`。
- 写入时机（主路径在**服务层**，以便拿到操作人）：
  - 新建物品 → 写一条初始日志：`null → 初始状态`。
  - `PATCH /api/items/[id]` 改了 `status` → 追加一条日志（带操作人，可附 `note`）。
- 兜底：`items` 上加一个 `AFTER UPDATE OF status` 触发器，当 `status` 实际变化且未由服务层补记时记一条（`changed_by_user_id` 记为 null），防止有人绕过 API 直接改库导致历史缺漏。
- 物品被删除时，日志随 `ON DELETE CASCADE` 一并清除。

### 完整性行为

- 删除被物品引用的**支出（expense）** → 被 `ON DELETE RESTRICT` 拦截；expenses 服务/接口需把数据库错误映射成友好提示："该支出仍有关联物品，请先改挂或删除物品"。
- 删除被物品占用的**场地区域（venue_item）** → 同样被 RESTRICT 拦截，场地页删除流程需友好报错（见第 0 期）。

### API 与服务层

- `GET /api/items` — 列表，支持筛选：`q`（名称/编号/序列号搜索）、`kind`、`status`、`floor_id`、`venue_item_id`、`responsible_person`。
- `POST /api/items` — 新建。
- `GET /api/items/[id]` — 单条详情，**返回时带上 `status_logs`**（按 `changed_at` 倒序）。
- `PATCH /api/items/[id]` — 更新；改 `status` 时由服务层写状态日志。
- `DELETE /api/items/[id]` — 删除（连带状态日志级联清除）。
- 服务层 `src/lib/items/service.ts`：返回 `ServiceResult<T>`，集中校验实物/虚拟规则（与数据库 CHECK 同义，但给出可读错误信息），并负责 `created_by_user_id` 与状态日志写入。
- 筛选类型 `src/lib/items/filter-types.ts`：参照 expenses 的 `filtersToParams` / `paramsToFilters`。

### 界面

- **侧边栏**：在「成本管理」分组 children 中新增 `物品管理`，图标用 Lucide `Package`。分组顺序：支出管理 · **物品管理** · 场地布置 · 财务预测。
- **`/items` 列表页** `src/app/[locale]/(app)/items/page.tsx`（参照 expenses 列表）：
  - 顶部筛选：搜索、类型、状态、楼层、区域、负责人。
  - 表格列：`item_code`、名称、类型、成本（支出名 + 金额，点击可跳支出）、位置（`楼·区`）、数量、状态、负责人、序列号。
  - 新增/编辑弹窗。
- **新增/编辑表单**：
  - 类型切换（实物 / 虚拟）。
  - **实物** → 展开必填：支出选择器（搜索现有 expenses，默认偏向 `tangible_asset` 类别但允许全部，显示名称/金额/日期）+ 放置位置级联（楼层 → 区域，区域取该楼层的 `venue_items`）。
  - **虚拟** → 隐藏放置位置；成本变选填。
  - 其余字段：数量、状态、负责人、序列号、备注。`item_code` 为系统生成、只读展示。
- **物品详情 / 状态时间线**：在编辑弹窗内（或抽屉）展示一块**状态变更时间线**——谁、何时、从什么状态改到什么状态、备注。

## 第 2 期：联动展示 + 照片

- **场地画布联动**：在场地页 `VenueInspector`（选中某区域时）增加"该区域物品"列表与**总成本**汇总；画布上区域可显示物品数量角标。数据来源 `GET /api/items?venue_item_id=...`。
- **照片上传**：新建 Supabase Storage 存储桶 `item-photos`，表单支持上传，`photo_url` 存可访问 URL。第 2 期前 `photo_url` 留空不用。
- **可选软校验**：同一支出下物品数量之和与支出数量不一致时，在表单/列表给出非阻断提示。

## i18n

- 导航：`nav.items` = `物品管理` / `Item Management`。
- 新增 `items.*` 命名空间：列表/表单/筛选/状态枚举/状态时间线/编号与序列号等文案，中英文同时维护。
- 状态枚举文案：`in_use`=使用中、`in_storage`=入库、`under_repair`=维修中、`disposed`=已报废。

## 测试

沿用现有 `node:test` + `assert` 风格（参考 `src/venue/layoutData.test.ts`）：

- `items` 服务：实物缺成本/缺位置应报错；虚拟带位置应报错；新建写初始状态日志；改状态追加日志。
- 编号生成：跨年重置、并发不重号（针对计数逻辑）。
- 完整性：删除被引用的支出/区域被拦截。
- 场地服务：`PUT /api/venue` 的 upsert + diff（新增/更新/删除）行为。

## 迁移与回滚要点

- 新迁移编号：场地 `029_venue.sql`、物品 `030_items.sql`（按 supabase/migrations 既有顺序，当前最新为 028）。
- 场地入库后保留 localStorage 兼容期：提供一次性导入，避免老用户布局丢失。
- 所有新表均可独立回滚（drop table），不影响 expenses 等既有数据。

## 实现顺序

第 0 期（场地入库）→ 第 1 期（物品台账，依赖第 0 期的 `venue_items`）→ 第 2 期（联动展示 + 照片）。每期之间设评审检查点。
