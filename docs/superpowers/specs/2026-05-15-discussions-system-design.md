# Discussions System Design

## Goal

Let users hold discussions anchored to business data on a page. Each discussion is an independent topic (not a row-level comment thread), bound to a service agent, and carries a globally unique `topic_code` so future email and IM channels can route messages back into the same thread.

The first version is a Web-only discussion system + service-agent routing skeleton. It is intentionally narrow:

- 3 high-frequency pages get discussion entry points: 费用 (expenses), 收入预测 (finance_forecast), 达人 (creators).
- Within those pages, users can discuss three kinds of subjects: a single record, a current filter result, or a saved view.
- Users see discussions by entering the relevant page, plus a lightweight "本页近期讨论" entry per page. There is no global inbox, no unread state, no notification center in v1.
- The schema and message model are forward-compatible with email/IM/agent-driven channels, but only the `web` channel is implemented in v1.

## Non-goals (v1)

- Global inbox, unread counters, notifications, mentions.
- Email or IM channel binding.
- Agent auto-summary, auto-close, action extraction.
- Attachments, reactions, edit history, message delivery state.
- External-contact identity, channel binding tables, full-text search.

## Core Concepts

**Thread = a topic, not a row.** A single expense row can have multiple threads:

```
费用 A
- EXP-2026-000001  为什么还没付款？
- EXP-2026-000002  金额是否录错？
- EXP-2026-000003  是否要分摊？
```

The page-level badge `[讨论 3]` reflects the count of threads under that subject, not the count of messages.

**Subject = what the thread is about.** Three subject types in v1:

- `record` — a single business row. `entity_id` identifies it.
- `filter` — the current filter result on a list page. `subject_hash` normalizes the filter so the same filter from different sessions matches.
- `saved_view` — a persisted named view. `entity_id` references the saved-view row.

**topic_code = cross-channel anchor.** Format `<PREFIX>-<YEAR>-<6-digit-seq>` (e.g. `EXP-2026-000001`). Generated by a database function inside the transaction that inserts the thread, so concurrent creates never collide. The prefix is stored on `service_agents.topic_prefix`, not hardcoded in the function.

**Service agent = who owns this discussion's domain.** Each `service_key` maps to exactly one agent via `service_agents`. v1 uses this only for display ownership; agent-driven summary is a v3 milestone.

## Data Model

### `service_agents` — service registry + agent routing

```sql
create table service_agents (
  service_key   text primary key,
  topic_prefix  text not null check (topic_prefix ~ '^[A-Z]{2,5}$'),
  agent_id      uuid not null references agents(id),
  display_name  text not null,
  is_active     boolean not null default true,
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now()
);
```

Doubles as a service registry. New services are added by `insert`, not by enum alteration. Other tables FK to `service_key` for DB-level validation without enum rigidity.

**v1 seed:**

| service_key | topic_prefix | agent | display_name |
|---|---|---|---|
| `expenses` | `EXP` | Morgan (Finance Agent) | 费用协作 Agent |
| `finance_forecast` | `FIN` | Casey (Growth Agent) | 财务预测 Agent |
| `creators` | `CRE` | Jordan (Ops Agent) | 达人协作 Agent |

The reuse of existing agents is a v1 expedient. Their `prompt_template` fields are still creator-individual oriented (Morgan outputs ROI JSON, etc.), which doesn't matter in v1 because no LLM call is made from a discussion. When agent summary lands in v3, either add a discussion-mode branch to each agent's template or seed sibling agents.

### `discussion_topic_sequences` — per-service per-year counter

```sql
create table discussion_topic_sequences (
  service_key         text not null references service_agents(service_key),
  year                int  not null,
  last_issued_number  bigint not null default 0,
  updated_at          timestamptz not null default now(),
  primary key (service_key, year)
);
```

Field name reflects actual semantics: it stores the most recently issued number. The generator function below uses `INSERT ... ON CONFLICT ... DO UPDATE` with `RETURNING` to claim a number atomically.

### `discussion_threads`

```sql
create type discussion_subject_type as enum ('record', 'filter', 'saved_view');
create type discussion_status       as enum ('open', 'resolved');

create table discussion_threads (
  id                  uuid primary key default uuid_generate_v4(),
  topic_code          text not null unique,
  service_key         text not null references service_agents(service_key),
  assigned_agent_id   uuid not null references agents(id),
  subject_type        discussion_subject_type not null,
  entity_type         text not null,
  entity_id           uuid,
  subject_hash        text,
  subject_payload     jsonb not null,
  title               text not null,
  status              discussion_status not null default 'open',
  created_by_user_id  uuid not null references users(id),
  resolved_by_user_id uuid references users(id),
  resolved_at         timestamptz,
  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now(),
  check (
    (subject_type = 'filter'                  and subject_hash is not null and entity_id is null) or
    (subject_type in ('record','saved_view')  and entity_id   is not null and subject_hash is null)
  )
);

create index idx_threads_record on discussion_threads
  (service_key, subject_type, entity_type, entity_id, status)
  where subject_type in ('record','saved_view');

create index idx_threads_filter on discussion_threads
  (service_key, subject_type, entity_type, subject_hash, status)
  where subject_type = 'filter';
```

- Same subject can have multiple threads — no uniqueness constraint beyond `topic_code`.
- Partial indexes keep the lookup paths separate for record/saved_view vs filter.
- `subject_payload` stores the human-readable label, source route, snapshot of filters, etc. Always present; UI reads from it to render thread cards without rejoining the source table.

### `discussion_messages`

```sql
create type discussion_sender_type as enum ('user', 'agent', 'external');
create type discussion_channel     as enum ('web', 'email', 'im');

create table discussion_messages (
  id              uuid primary key default uuid_generate_v4(),
  thread_id       uuid not null references discussion_threads(id) on delete cascade,
  parent_id       uuid references discussion_messages(id),
  sender_type     discussion_sender_type not null,
  sender_user_id  uuid references users(id),
  sender_agent_id uuid references agents(id),
  channel         discussion_channel not null default 'web',
  body            text not null,
  metadata        jsonb not null default '{}'::jsonb,
  created_at      timestamptz not null default now(),
  updated_at      timestamptz not null default now(),
  deleted_at      timestamptz,
  check (
    (sender_type = 'user'  and sender_user_id  is not null) or
    (sender_type = 'agent' and sender_agent_id is not null) or
    (sender_type = 'external')
  )
);

create index idx_messages_thread on discussion_messages (thread_id, created_at)
  where deleted_at is null;
```

- `parent_id` supports lightweight reply threading. v1 UI renders a single flat stream; replies show as quoted citations.
- `channel` defaults to `web`; future email/im inbound writes use the same table with different channel values.
- `metadata` reserved for external message IDs, email-thread IDs, IM room IDs, summary types, action extractions.
- Soft delete only — `deleted_at` set instead of `DELETE`.

### topic_code generator

```sql
create or replace function next_discussion_topic_code(p_service_key text)
returns text language plpgsql as $$
declare
  v_year   int := extract(year from now())::int;
  v_num    bigint;
  v_prefix text;
begin
  select topic_prefix into v_prefix
  from service_agents
  where service_key = p_service_key and is_active;
  if v_prefix is null then
    raise exception 'unknown or inactive service_key: %', p_service_key;
  end if;

  insert into discussion_topic_sequences (service_key, year, last_issued_number)
  values (p_service_key, v_year, 1)
  on conflict (service_key, year) do update
    set last_issued_number = discussion_topic_sequences.last_issued_number + 1,
        updated_at = now()
  returning last_issued_number into v_num;

  return format('%s-%s-%s', v_prefix, v_year, lpad(v_num::text, 6, '0'));
end $$;
```

Concurrency safety relies on the row-level lock taken by `ON CONFLICT DO UPDATE`. Two concurrent inserts on the same `(service_key, year)` serialize on the row.

## Subject Normalization

Lives in `src/lib/discussions/subject.ts`. The frontend never computes hashes — it sends raw subject descriptors, the server normalizes them.

```ts
type SubjectInput =
  | { subjectType: 'record';     serviceKey: string; entityType: string; entityId: string; label: string; route: string }
  | { subjectType: 'filter';     serviceKey: string; entityType: string; filters: Record<string, unknown>; label: string; route: string }
  | { subjectType: 'saved_view'; serviceKey: string; entityType: string; entityId: string; label: string; route: string }
```

For `filter` subjects the server:

1. Drops `null` / `undefined` / empty-array values.
2. Keeps only keys in the per-entity_type whitelist (`FILTER_WHITELIST[entityType]`).
3. Normalizes values: dates → ISO yyyy-mm-dd, enums lowercased, array values sorted.
4. Sorts object keys recursively to produce a canonical JSON string.
5. SHA-256 hex of the canonical JSON → `subject_hash`.
6. Writes the normalized filters into `subject_payload.filters`.

The whitelist guarantees subject-hash stability across UI refactors: only meaningful filter keys participate. For `expenses` v1 the whitelist is:

```
status, date_from, date_to, category_id, creator_id,
currency, amount_min, amount_max, tags
```

## Permissions

`src/lib/discussions/permissions.ts` exports `canReadThread(actor, thread)` and `canResolveThread(actor, thread)`. **Both must be invoked on every read path** — detail, list, messages, resolve, **and counts**. Counts especially: leaking a count of threads on a private saved view leaks the view's existence.

```ts
function canReadThread(actor, thread) {
  if (actor.is_admin) return true
  if (thread.subject_type === 'record') return true   // follows business-row visibility
  if (thread.subject_type === 'filter') return true   // filter criteria carry no sensitive info
  if (thread.subject_type === 'saved_view') {
    if (thread.entity_type === 'expense_saved_view')    return view.user_id === actor.id
    if (thread.entity_type === 'finance_forecast_view') return view.is_public || view.owner_id === actor.id
  }
  return false
}
```

`record` subjects defer to the underlying table's existing access control. When those tables tighten, this layer tightens automatically. `saved_view` is where v1 locks down hardest because it is the highest-risk leak vector.

`canResolveThread` is more restrictive:

```ts
function canResolveThread(actor, thread) {
  if (actor.is_admin) return true
  return thread.created_by_user_id === actor.id
}
```

Agents do not auto-close threads in v1.

## API Surface

All routes live under `src/app/api/discussions/`.

| Method | Path | Purpose |
|---|---|---|
| POST | `/api/discussions/threads` | Create a thread. Body: `{ subject, title, firstMessage }`. Generates `topic_code`, inserts thread + first message in one transaction. |
| GET | `/api/discussions/threads?...` | List threads for a subject. Query: `serviceKey`, `entityType`, plus either `entityId` or raw filter (server normalizes), and optional `status`. Results filtered by `canReadThread`. |
| GET | `/api/discussions/threads/:id` | Thread detail + messages. 403 if `canReadThread` denies. |
| POST | `/api/discussions/threads/:id/messages` | Append a message. Body: `{ body, parentId? }`. |
| PATCH | `/api/discussions/threads/:id/resolve` | Mark resolved. Gated by `canResolveThread`. |
| POST | `/api/discussions/subject/resolve-counts` | Batch counts. Body: `{ subjects: SubjectInput[] }`. Returns `[{ key, openCount, resolvedCount }]` after permission filtering. Critical for the expenses table where every row needs a badge — without batching this is O(rows) requests. |

DELETE message and agent-summary endpoints are deferred to PR3.

## Frontend Integration

Components in `src/components/discussions/`:

- `DiscussionBadge` — renders `[讨论 N]` / `[已结束 1]` / `[讨论]` based on counts from context.
- `DiscussionPanel` — right-side drawer, the primary UI shell.
- `ThreadList` — when a subject has multiple threads, list them first.
- `ThreadView` — header (topic_code, subject summary, agent owner, status), message stream, input box, "结束讨论" button.
- `PageDiscussionsEntry` — top-of-page strip showing recent open threads scoped to this page's `service_key`. Cheap discovery mechanism; not a global inbox.
- `DiscussionContext` — page-level provider that batches counts via `resolve-counts` and exposes a `useDiscussionCount(subject)` hook for badges.

The expenses list page is the v1 reference integration:

```
[本页近期讨论 3]                          (PageDiscussionsEntry)

筛选条件: [状态: 未付] [金额: > 1000]   [讨论 2]    (filter-subject badge)

┌────────────┬───────┬──────────┬─────────┬──────────┐
│ 日期       │ 类别  │ 金额     │ 状态    │          │
├────────────┼───────┼──────────┼─────────┼──────────┤
│ 2026-05-01 │ 云服务│ ¥3,200   │ 未付    │ [讨论 3] │  <- record-subject badge
```
