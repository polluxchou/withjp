# 工作总结 · 2026-05-11

## 主题：设备管理 → 支出管理（Expense Management）升级

---

## 一、背景

原有的"设备管理"模块（`/devices`）仅覆盖有形资产采购，无法反映团队的完整运营支出（薪资、租金、差旅、耗材、云服务等），导致财务视图不完整、成本侧数据缺失。

本次升级将其扩展为统一的**支出管理（Expense Management）**模块，覆盖所有运营成本类别，成为团队对账、预算追踪、P&L 分析的基础数据源。

---

## 二、产品设计

### 支出类别

| 类别 Key | 中文名 | 典型支出项 |
|---------|--------|-----------|
| `tangible_asset` | 有形资产 | 设备、硬件（原 devices 数据归此类） |
| `salary` | 薪资成本 | 工资、外包、劳务费 |
| `rent` | 租金 | 办公室、仓库、场地 |
| `travel` | 差旅费 | 交通、住宿、餐饮 |
| `office_supplies` | 办公耗材 | 文具、打印耗材 |
| `cloud_services` | 云服务/网络 | SaaS、API 调用、宽带、域名 |

### 字段动态规则（按类别）

| 字段 | tangible_asset | salary | rent | travel | office_supplies | cloud_services |
|------|:-:|:-:|:-:|:-:|:-:|:-:|
| 单价 × 数量 | ✅ | ❌ 仅总额 | ❌ 仅总额 | ❌ 仅总额 | ✅ | ❌ 仅总额 |
| 归属周期 period | ❌ | ✅ | ✅ | ❌ | ❌ | ✅ |
| 地点 location | ✅ | ❌ | ✅ | ✅ | ❌ | ❌ |

### 支付方式（从自由文本升级为枚举）

| 选项值 | 显示名 | 约束 |
|--------|--------|------|
| `company_account` | 公司公共账户 | 经办人必须为 `with-new` / `JP-代理陈昊` / `JP-代理小兽` 三选一 |
| `wechat_pay` | 微信支付 | 无额外约束 |
| `alipay` | 支付宝 | 无额外约束 |
| `bank_card` | 银行卡 | 无额外约束 |

### 历史数据兼容策略

- 旧 `devices` 数据整体迁移至 `expenses`，自动标记为 `tangible_asset`
- 旧自由文本支付方式保存至 `payment_method_legacy` 字段，编辑时展示橙色提示，用户选择新枚举值后自动清空

### 页面 KPI 看板

- **总支出**（所有记录合计）
- **已付款**（`paid` 状态合计）
- **预算/待付款**（`budgeted` + `ordered_unpaid` 合计）
- **本月支出**（当月 `paid` 合计）

### 图表

- **类别占比**：横向进度条，各类别总额及百分比
- **累计趋势**：折线图，已付款 / 已预算 / 待付款三线，支持月/季/年粒度切换

## 三、技术架构

### 数据库（Migration 011）

```sql
-- 新增枚举
create type expense_category as enum (
  'tangible_asset', 'salary', 'rent', 'travel', 'office_supplies', 'cloud_services'
);

create type expense_payment_method as enum (
  'company_account', 'wechat_pay', 'alipay', 'bank_card'
);
```

**`expenses` 表核心字段：**

| 字段 | 类型 | 说明 |
|------|------|------|
| `expense_category` | enum | 支出类别 |
| `item_name` | text | 支出名称（原 device_name） |
| `unit_price` × `quantity` → `total_price` | numeric | generated always as stored |
| `expense_date` | date | 支出日期（原 purchase_date） |
| `location` | text | 地点（原 purchase_location） |
| `purpose` | text | 用途（原 purchase_purpose） |
| `period` | text | 归属周期，如 `2025-05`，用于薪资/租金/云服务 |
| `payment_method` | enum \| null | 新枚举支付方式 |
| `payment_method_legacy` | text \| null | 保留旧自由文本值 |
| `payment_status` | text | 5种状态，check constraint 校验 |

**数据库级约束：**
```sql
constraint expenses_company_account_buyer check (
  payment_method is distinct from 'company_account'
  or buyer_name in ('with-new', 'JP-代理陈昊', 'JP-代理小兽')
)
```

数据迁移：`insert into expenses ... select ... from devices`，历史数据零丢失。

### API 设计

```
GET    /api/expenses          查询列表（q, category, payment_status, payment_method,
                              user_name, buyer_name, date_from, date_to, period）
POST   /api/expenses          新增支出（含公司账户经办人校验）
PATCH  /api/expenses/:id      编辑支出（自动清空 payment_method_legacy）
DELETE /api/expenses/:id      删除支出
```

### 前端表单动态逻辑

```
选择类别 → 动态显示/隐藏：单价×数量、归属周期、地点字段

选择其他支付方式 → 经办人为自由文本 <input>
编辑历史记录且有 payment_method_legacy → 顶部显示橙色提示横幅
```
